# Novel Features Brainstorm — dominos-pp-cli (reprint)

> Audit trail. Generated by the novel-features subagent per Phase 1.5 Step 1.5c.5.

## Customer model

**Maya — the Friday-night reorderer.** A Seattle apartment-dweller with a four-week-running pattern: large pepperoni with extra cheese, garlic knots, a 2L Sprite, delivered to 421 N 63rd St, billed to the same Visa. Maya used to use Domino's "Easy Order" feature on the website but lost it when she switched accounts; now she rebuilds the cart from order history every Friday around 7pm.

*Today (without this CLI):* Opens dominos.com, signs in, scrolls Order History, clicks "Reorder" on the most recent matching order, manually fixes whichever item Domino's silently dropped (the menu rotates), enters CVV, places.
*Weekly ritual:* Friday 7pm reorder of "the usual" with a tracker tab open until the doorbell rings.
*Frustration:* "I order the same thing every week. Six clicks, two refreshes, and one 'this item is no longer available' surprise per order. Why isn't this one command?"

**Devon — the deal hunter.** Orders Domino's two or three times a month and refuses to pay rack rate. Knows that the deals page has 12+ promotions, that loyalty members get a separate set, and that some deals stack (e.g. Mix & Match + a Points reward) but the website never tells you which combination is cheapest.

*Today (without this CLI):* Loads the deals page in one tab, builds the cart in another, eyeballs which promo applies, manually swaps codes in checkout, occasionally walks away when the math doesn't beat ordering elsewhere.
*Weekly ritual:* Pre-order coupon hunt: 3-5 minutes of mental cross-reference between the cart and the active deals before committing.
*Frustration:* "There is no view that says 'apply every applicable deal to this exact cart and tell me the lowest total.' I have to be the optimizer."

**Priya — the party planner / office orderer.** Coordinates Friday team lunches for 12-25 people across an engineering org. Some weeks she splits the order across two stores because her closest one quotes 75-minute waits.

*Today (without this CLI):* Spreadsheet of names and pizza preferences, calls the store, dictates each pie, repeats per store when she has to split. Wait time is a guess until she hears it on the phone.
*Weekly ritual:* Thursday afternoon survey, Friday morning store-pick + multi-cart build, Friday 11:30am submit, then babysit the trackers.
*Frustration:* "Every order goes through one store. Nothing tells me the wait times across the three stores in radius, and nothing splits a 25-pie order across two of them."

**Ace — the agent / automation power user.** Runs a homelab Claude that does odd jobs at the end of standup ("order team lunch, deliver to the office, budget $120, reuse last week's order"). Wants tools that exit cleanly with structured output.

*Today (without this CLI):* Brittle Puppeteer against dominos.com that breaks every time the checkout flow shifts. No reliable way to get a structured "did the order succeed" signal.
*Weekly ritual:* End-of-standup or end-of-sprint trigger that places a previously-validated order against a known address, then reports back the tracker URL.
*Frustration:* "There's no tool that takes a budget + address + cart template and returns `{order_id, eta, total}` as JSON. Everything is HTML scraping."

## Survivors and kills

### Survivors

| # | Feature | Command | Score | Persona served | How It Works | Evidence |
|---|---------|---------|-------|---------------|-------------|----------|
| 1 | Named order templates | `template save` / `template order <name>` | 9/10 (DF 3, UP 3, BF 2, RB 1) | Maya, Ace | SQLite `templates` table stores serialized cart; `template order` rehydrates and runs validate→price→place | Brief Top Workflow #5; apizza has cart persistence but no named templates; prior built |
| 2 | Smart reorder with substitution | `reorder --last --substitute-unavailable --dry-run` | 8/10 (DF 3, UP 2, BF 2, RB 1) | Maya | Joins local `orders` (auth-synced history) with current `menu_items` FTS5 index; ranked substitution by name+category similarity | Brief mentions menu rotation; competing wrappers have neither order history nor full local menu; prior built |
| 3 | Deal optimizer (with stacking) | `deals best --cart <name> [--stack]` | 8/10 (DF 3, UP 3, BF 1, RB 1) | Devon, Ace | For each deal in DealsList (and loyalty rewards if logged in), call price-order with the deal applied; rank by total. `--stack` enumerates 2-3-deal combinations | Devon's frustration verbatim; CheckDraftDeal is per-deal; prior built |
| 4 | Cross-store price comparison | `compare-prices --address ... --items ...` | 7/10 (DF 2, UP 2, BF 2, RB 1) | Devon, Priya | For each store from store-locator, build identical cart and call price-order; rank by total (incl. delivery fee) | Brief Product Thesis cites this as "impossible without local data layer"; prior built |
| 5 | Multi-store wait-time scoreboard | `stores wait --address ...` | 7/10 (DF 3, UP 2, BF 1, RB 1) | Priya, Maya | For each store in radius, fetch CartEtaMinutes (GraphQL BFF op) with a small dummy cart; sort by ETA | Priya's frustration verbatim; CartEtaMinutes is reverse-engineered and absent from every wrapper |
| 6 | Live delivery tracker with polling | `track --phone ... --watch --interval 30s` | 8/10 (DF 3, UP 3, BF 2, RB 0) | Maya, Ace | Poll GetTrackerData every interval; emit one JSON line per status transition; exit 0 on delivered | Brief explicitly names this persona; node-dominos has single-shot tracker only; prior built |
| 7 | One-shot agent order | `order quick --template <name> --eta-watch --json` | 7/10 (DF 2, UP 2, BF 2, RB 1) | Ace, Maya | Composes template-load → validate → price → place (gated on `--confirm`) → track --watch; emits final JSON | Ace persona explicitly asks for "agent-shaped tool"; user vision calls out agent-shaped output |
| 8 | Cart-shaped deal eligibility | `deals eligible --cart <name>` | 6/10 (DF 2, UP 2, BF 1, RB 1) | Devon | Pulls DealsList; for each deal, locally checks deal predicates against cart contents; tags reasons-for-fail | Devon's stack-versus-eligibility confusion; DealsList exposes deal predicates but UI hides which match |

### Killed candidates

| Feature | Kill reason | Closest surviving sibling |
|---------|-------------|---------------------------|
| Split-order across stores (C6) | Scope creep — multi-cart orchestration + CSV ingestion + cross-store inventory check is two features pretending to be one | Cross-store price comparison (#4) |
| Spending analytics (C8) | No named persona reaches for this weekly | Smart reorder with substitution (#2) |
| Menu diff (C9) | No persona named in this brief tracks menu changes; "power users want to know" is speculative | Cart-shaped deal eligibility (#8) |
| Nutrition calculator (C10) | No persona; one-shot read; menu API already returns nutrition — `cart show --json \| jq` does this | None — drop entirely |
| Store health score (C11) | "Composite health" is hand-wavy; useful ingredient is wait-time, which #5 exposes directly | Multi-store wait-time scoreboard (#5) |
| Deal-stack as separate feature (C12) | Same pipeline as deal optimizer; folded in as `--stack` flag on (#3) | Deal optimizer (#3) |
| Loyalty redemption planner (C15) | Niche; one-shot calculation; doesn't compound across days | Deal optimizer (#3) |

| Prior feature | Verdict | Justification |
|---|---|---|
| Cross-store price comparison (`compare-prices`) | **Keep** | Devon + Priya both reach for it; command name preserved; scores 7/10 |

| Spending analytics (`analytics`) | **Drop** | None of the four named personas runs this weekly |
| Live delivery tracker (`tracking --watch`) | **Reframe** (rename `tracking` → `track`) | Persona fit is strong; rename to match brief and recipes |
| Smart reorder with substitution (`reorder`) | **Keep** | Maya's "menu rotated and dropped one item" frustration; command preserved; scores 8/10 |
