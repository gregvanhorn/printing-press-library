# Cal.com CLI Brief

## API Identity
- Domain: Scheduling infrastructure (open-source Calendly alternative)
- Users: Developers building scheduling into apps, teams managing availability, solo professionals managing bookings, AI agents automating booking flows
- Data profile: 285 operations across 181 paths. Resources: bookings (24 paths), event-types, schedules (3), slots (5 with reserve flow), calendars (17), teams (20), organizations (78), conferencing (7), webhooks, stripe, OOO, routing forms
- Base URL: https://api.cal.com/v2
- Auth: Bearer token (`cal_live_*` prefix), OR managed-user OAuth access token. Token format inferred from header parameters; `bearerAuth` is declared in `components.securitySchemes`.
- Per-endpoint versioning: `cal-api-version` header required on 27/285 ops with different values per resource group: Bookings `2024-08-13`, Event Types `2024-06-14`, Schedules `2024-06-11`, Slots `2024-09-04`. Generator must emit per-endpoint version.
- Rate limits: 120 req/min (API key), 500 req/min (OAuth)
- v1 deprecated April 8, 2026 — v2 is the only path forward

## Reachability Risk
- **None** — live key (`cal_live_*`) probed against `/v2/me`, `/v2/event-types`, `/v2/bookings`, `/v2/schedules` all returned 200 with real data. API is actively maintained.

## Top Workflows
1. **End-to-end booking** — find available slot for an event type, optionally reserve, create booking with attendee details, confirm. Today the official MCP requires the LLM to chain these.
2. **Booking lifecycle** — list, get, reschedule, cancel, confirm, decline, mark no-show, reassign, manage attendees and guests.
3. **Event type configuration** — create/update event types with duration, scheduling rules, hosts. Round-robin team event types.
4. **Availability management** — schedule CRUD with weekly slots, query available slots for date range across one or multiple event types.
5. **Calendar sync** — connect external calendars (Google, Outlook, Apple/CalDAV), manage busy times, destination calendar routing, selected-calendars for availability.
6. **Team & org administration** — create teams, manage memberships and roles, verified resources, conferencing apps, routing forms, delegation credentials, custom user attributes.
7. **Webhooks** — register triggers (BOOKING_CREATED/CANCELLED/RESCHEDULED, MEETING_ENDED, etc.) per event type or org-wide, manage signing secrets.

## Codebase Intelligence
- Source: gh + WebFetch on calcom/cal-mcp + bcharleson/calcom-cli + apps/api/v2 in calcom/cal.com
- Auth: header `Authorization: Bearer <token>`, env var `CAL_API_KEY` is the de facto convention. Token prefix `cal_live_` for production, `cal_test_` for sandbox. OAuth managed users have a separate refresh flow at `/v2/oauth/{clientId}/refresh`.
- Data model: top-level entities Booking, EventType, Schedule, User (Me), Team, Calendar, Webhook. Booking has nested attendees + guests + references + recordings + transcripts. EventType has nested hosts + locations + bookingFields.
- Rate limiting: declared in headers (`X-RateLimit-*`), 429 on exceedance.
- Architecture: NestJS controllers under `apps/api/v2/src/modules/*` and `ee/*` (enterprise). The Cal.com team generates Swagger only in dev — no public spec URL. The cached spec we have is the most authoritative open snapshot.
- Composed intents: NONE in any official tool. cal-mcp and bcharleson/calcom-cli both expose 1:1 endpoint mirrors. The big competitive opening.

## Ecosystem Inventory
- **bcharleson/calcom-cli** (npm, TypeScript, 14 groups, 61 tools, dual CLI+MCP) — closest competing CLI. No local cache, no composed intents, no dry-run, no batch input, has `--fields`/`--quiet`/`--pretty`. Created 2026-03-19 (active).
- **calcom/cal-mcp** (official, 167 endpoint mirrors, 9 default tools) — DORMANT 11 months since 2025-05-22. Auto-generated by Buildwithlayer from OpenAPI. No composed intents.
- **Composio Cal toolkit** (~140 hosted actions) — the maximal feature checklist. Active.
- **dsddet/booking_chest** (Python MCP, ~70 v2 tools across 10 modules) — second-best v2 coverage outside vendor.
- **aditzel/caldotcom-api-v2-sdk** (TypeScript SDK, 12 resource groups, 3 auth modes inc. OAuth refresh) — active 2025-12-10.
- **vinayh/calcom-mcp** (12 tools, .mcpb), **Danielpeter-99/calcom-mcp** (8 tools, FastMCP), **mumunha/cal_dot_com_mcpserver** (4 tools, v1 only, dormant) — small.
- **terrylica/cc-skills calcom-commander** — Claude Code plugin built on Bash+Bun+Docker; closest "agent CLI" but specialized to Telegram/Pushover notifications and infra deploy.

## Data Layer
- Primary entities to mirror in SQLite: bookings, event_types, schedules, teams, users, webhooks, calendars (connected + selected), conferencing_apps, ooo_entries
- Sync cursor: bookings have `afterStart`/`status`/`updatedAfter` filters → time-windowed sync. Event types and schedules are relatively static → full refresh.
- FTS/search: bookings (by title, attendee name, notes), event types (by title/slug/description), teams (by name), webhooks (by subscriber URL/triggers).
- Compound queries that no single API call exposes:
  - Booking density per weekday/hour
  - Cancellation/no-show rates per event type, per attendee, over time window
  - "Next available slot across N event types" — slot search across multiple types
  - Conflicting busy-time overlap with active bookings
  - Webhook trigger coverage gaps (which lifecycle events are not subscribed?)

## User Vision
- The user explicitly said "we previously printed this but on a very old version" and gave full read+write API key. They expect a complete redo with current Printing Press capabilities (scoring v2, MCP intent surface, store-backed transcendence). They've authorized real bookings during dogfood, including visiting booking links to see the effect — meaning Phase 5 should exercise a full create→list→reschedule→cancel cycle, not just dry-runs.

## Product Thesis
- Name: cal-com-pp-cli (binary), `~/printing-press/library/cal-com/`
- Why it should exist: Cal.com has 9+ competing tools, none of which combine (a) Go single binary with no runtime deps, (b) local SQLite mirror enabling offline queries, agenda views, and analytics, (c) composed intents that bundle slot-find + reserve + book + confirm into one safe command with `--dry-run`, (d) auth doctor diagnostics across api-key + OAuth managed-user modes, (e) cross-event-type slot search, (f) webhook trigger coverage gap detection. Every existing tool is a flat endpoint mirror. We can be the agent-native, offline-first, intent-driven Cal.com CLI.

## Build Priorities
1. **Foundation:** data layer for bookings, event-types, schedules, teams, webhooks, calendars + sync (with `--full` and incremental cursor) + FTS search
2. **Absorb:** match every command in bcharleson/calcom-cli (61) and the high-value Composio actions (mark-no-show, reassign, decline-with-reason, OOO CRUD, calendar busy-times, selected-calendars toggle, webhook CRUD with trigger constants, slot reserve flow) — covers >90% of the field
3. **Transcend:** composed `book` intent (find→reserve→create→confirm), `today`/`week`/`agenda` offline views, `slots find --any-of` cross-event-type search, `analytics` (no-show, cancellation, booking density), `webhooks coverage` (trigger gap detection), `conflict-check` (busy-time vs active bookings), `auth doctor` (diagnose API key + OAuth state)
